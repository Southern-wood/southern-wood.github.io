前言#

我曾经在一篇博客中总结我使用 Linux 服务器处理并行任务的经验：如何在共享的 Linux 服务器上合理使用资源，那时，我主要是用 shell scripts 和 nohup 来管理并行任务。

最近，我们的课题已经完成了大部分的工作，进入了最后的补充实验阶段，即全面评估各种 baseline 在不同数据集、不同场景设置下的性能。

这个过程中的任务数量非常庞大，我们至少有七到八种不同的 baseline，需要在两到三个数据集上评估，每种数据集又存在十余种不同的场景设置，甚至涉及到训练阶段和测试阶段的场景设置不同的情况。排列组合后，我们有成百上千个子任务需要完成。

我不得不承认，纯粹使用 shell 脚本和 nohup 的组合已经无法解决这样庞大的并行任务了，以下是我真实遇到的一些的问题：

shell scripts 难以区分 nohup 进程的状态#

nohup 能够守护进程，但是它将切断 shell 脚本和后台进程之间的连接，这意味着如果某个进程崩溃了而不是正常结束，简单的 shell 脚本就无法识别这种情况了，更不能自动重启进程或者做其他的补救措施。

想象一下，如果乘着夜深人静、服务器空闲的时候，你启动了一个 shell scripts，里面有十几个 nohup 进程会按顺序执行。第二天起床一看，所有的输出文件都是 RuntimeError:CUDA error:out of memory…原来昨晚课题组有人跑了个 LLM 的微调任务。

🚨 CUDA OOM NIGHTMARE 🚨

这时候你会无比希望有一个自动重试任务的功能。

孤儿进程的衍生问题#

同样因为 nohup 的原因，如果运行过程中 shell scripts 本身崩溃了或者被手动终止了，那么 nohup 启动的任务会被转移到 init 进程下成为 “孤儿进程”。这本身就是 nohup 存在的意义，但是这个特性有时候很让人恼火。

想象下面这个场景：

假如你对一些代码做了修改，启动了并行的脚本之后，你才意识到有些错误没解决，或者需要修改些参数；这时候你只是 ctrl+c 终止了 shell scripts 本身，而忘记了杀死所有已启动的后台进程；你做了必要的修改之后启动了新的脚本，美滋滋地上床睡觉了。第二天起来却发现输出文件混在一起，根本无法分辨哪些是原始的错误进程，哪些是修改后的正确进程输出，一夜白忙。

而即使我时刻记得需要杀死 nohup 进程，事情也没这么简单：为了方便管理和并行脚本之间的迁移，我统一了 baseline 的代码，现在所有的任务都以 main.py 为入口文件，接受相同的几个参数用于设置场景，因此启动不同进程的命令之间是高度相似，甚至完全相同的。这意味着区分需要杀死的进程非常复杂，pkill -f 不能完成这个任务，而必须按照启动时间来区分进程，手动 kill 对应的 PID。

监控不同 baseline 的任务进度#

nohup 能守护进程，但是它不能守护启动它进程的 shell 脚本，解决这个问题的办法是再使用一个 nohup 来守护 shell 脚本本身。然而，这意味着 shell 脚本的输出也需要重定向到文件中，需要查看各个 baseline 的总体进度时，必须逐个检查文件，这个过程相当地繁琐并且无聊。

Tmux & Parallel#

当然，命令行工具的潜力是无穷的，只要你愿意，你可以仍然使用 shell scripts 和 nohup 来解决这些问题。只要不断组合不同的工具和命令，你总能实现一个功能丰富的并行化脚本，解决一切你遇到的问题。但这绝非我的风格，我不希望把一个简单的 shell scripts 写得像一个迷你操作系统。
